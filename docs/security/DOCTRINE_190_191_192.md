## §190 — pgcrypto Column Encryption Standard

**Purpose.** Standard pattern for at-rest column encryption of tenant-scoped secrets (webhook URLs, OAuth refresh tokens, provider webhook signing secrets, root-tier API credentials).

**Reference implementation:** `public.tenants.automation_webhook_url_encrypted` (Ship #1).

### The pattern (non-negotiable)

1. **Column shape.** `bytea NULL` column with `_encrypted` suffix. Plaintext column MUST NOT co-exist on the same row.
2. **Key material.** Symmetric key stored in `_internal_secrets` (one row per named key). Key value is generated via `secrets--generate_secret` (≥64 chars) and seeded once via a one-shot `seed-internal-secret` edge function. Never checked into source. Never returned from any RPC.
3. **Encrypt path.** `pgp_sym_encrypt(plaintext, key)` inside a `SECURITY DEFINER` admin-only setter RPC. Setter validates caller is platform owner or tenant admin scoped to the target row.
4. **Decrypt path.** `pgp_sym_decrypt(ciphertext, key)` inside a **separate** `SECURITY DEFINER` getter RPC. Getter is the **sole** plaintext path. Direct `SELECT` on the `_encrypted` column returns bytea only.
5. **Access log.** Every getter invocation writes to `pii_access_log` with `{actor_id, target_tenant_id, column_name, purpose}`. No exceptions — including admin reads.
6. **Grants.** `REVOKE SELECT` on the `_encrypted` column from `authenticated` and `anon`. `service_role` retains full access. Getter/setter RPCs `GRANT EXECUTE` to `authenticated` only.
7. **RLS second wall.** Table-level RLS restricts row visibility to the owning tenant + platform owner even though the column itself is unreadable — belt and suspenders.
8. **Catalog.** Getter and setter both land in `SECURITY_DEFINER_CATALOG.md` as Category B with the standard justification block.

### Rotation schedule (matches Cryptographic Key Management Policy)

| Credential class | Rotation cadence | Trigger events |
|---|---|---|
| Scoped credentials (API keys, OAuth tokens, service accounts) | **Annual** | Personnel change with access · suspected compromise · vendor-forced |
| Root-tier credentials (master passwords, admin auth tokens) | **Biennial** | Personnel change with access (immediate) · suspected compromise (immediate) · vendor-forced |
| pgcrypto column keys (`_internal_secrets` entries) | **Biennial** | Any getter/setter audit anomaly · key material suspected exposed |

Rotation SOP: (a) generate new key via `secrets--generate_secret`, (b) run one-shot re-encrypt migration that decrypts with old key and re-encrypts with new, (c) archive old key row with `rotated_at` timestamp, (d) verify decrypt path against new key, (e) log rotation to `paige_audit_log`.

## §191 — Automation Dispatch Envelope Contract

**Purpose.** Stable, versioned payload contract for every platform-emitted automation event so tenant orchestrators (n8n, LangGraph, Zapier, custom webhooks) can build against one shape regardless of trigger source.

### Envelope (required on every dispatch)

```jsonc
{
  "envelope_version": "1",
  "event_id": "uuid",              // idempotency key — dispatcher retries reuse this
  "event_kind": "stage_change",    // see registry below
  "tenant_id": "uuid",
  "rule_id": "uuid | null",        // null for system-emitted events
  "dispatched_at": "2026-07-01T17:45:00Z",
  "consent_status": {
    "required": "transactional | marketing | none",
    "granted": true,
    "checked_at": "2026-07-01T17:45:00Z"
  },
  "subject": { /* typed block, see registry */ }
}
```

### Event kind registry (typed subject blocks)

| `event_kind` | Subject block | Emitter | Ship |
|---|---|---|---|
| `stage_change` | `{deal_id, contact_id, from_stage_id, to_stage_id, pipeline_id, compose_intent}` | `on_deal_stage_change` trigger + `dispatch-stage-automation` | #1 (live) |
| `readiness_scan` | `{contact_id, scan_type, score, prior_score, delta}` | Scheduled proposals cron | #2 |
| `enrichment_completed` | `{contact_id, provider, fields_updated[], confidence}` | Enrichment pipeline | #3 |
| `calendar_booking` | `{booking_id, contact_id, staff_id, starts_at, ends_at, meeting_type}` | Google Calendar sync | Phase 2 |
| `payment_received` | `{invoice_id, contact_id, amount_cents, currency, provider}` | Stripe webhook handler | future |
| `document_signed` | `{envelope_id, contact_id, agreement_id, signed_at}` | Signature webhook | future |

### Rules

1. **Additive-only.** New fields on `subject` require no version bump. Removals or renames require `envelope_version` increment + parallel dispatch period.
2. **Consent gate.** Dispatcher MUST populate `consent_status` and MUST NOT emit when `granted=false` for a required category. Skipped events log to `stage_automation_events.status='skipped_consent'`.
3. **Idempotency.** `event_id` is stable across retries. Tenant orchestrators dedupe on it.
4. **Contract test.** Every new `event_kind` ships with a fixture JSON in `docs/security/dispatch_envelope_fixtures/` that CI validates against the shared JSON schema.

## §192 — Scoped Credentials Over Root Credentials

**Rule.** Every new third-party integration uses scoped, revocable credentials (API keys, OAuth tokens, service accounts) rather than root-level credentials (master account passwords, admin auth tokens) wherever the vendor supports both.

### Reference

| Vendor | Scoped (preferred) | Root-tier (avoid for new integrations) |
|---|---|---|
| Twilio | `TWILIO_API_KEY` (gateway-routed, this project as of 2026-07-01) | `TWILIO_ACCOUNT_SID` + `TWILIO_AUTH_TOKEN` |
| Stripe | Restricted API keys per surface (webhook, read, checkout) | `STRIPE_SECRET_KEY` full-access |
| Resend | Per-domain scoped keys | Account-wide send keys |
| Google (per-tenant) | OAuth 2.0 refresh tokens with narrowest scope set (`calendar.events.owned`, `userinfo.email`) | Service account with domain-wide delegation |
| Anthropic | Workspace-scoped keys | Organization root keys |
| n8n | Workflow-scoped API key when supported | Instance admin token |

### Applies to

- All **new** third-party integrations from doctrine landing forward.
- Existing integrations MAY migrate opportunistically. Migration slots are tracked in the Phase B queue.

### Migration follow-ups (queued)

- **TWILIO_AUTH_TOKEN** → migrate to `_internal_secrets.twilio_auth_token_encrypted` under §190 pattern (root-tier warrants column encryption). `TWILIO_API_KEY` stays in edge function env vars (scoped, gateway-routed, lower blast radius).
- Old Twilio-signed webhook verification (`handle-inbound-sms`) continues to use `TWILIO_AUTH_TOKEN` until migration — signature verification requires the same secret Twilio uses to sign, which is the account auth token. Retire alongside root-token migration.

### Rotation (see §190 table)

Scoped: annual. Root-tier: biennial + immediate on personnel change with access.
