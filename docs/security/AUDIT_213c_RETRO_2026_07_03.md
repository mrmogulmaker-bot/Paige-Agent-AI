# §213.c Retro-Audit — DO-Block Snapshot Isolation

**Date:** 2026-07-03
**Scope:** All 332 migrations under `supabase/migrations/`
**Doctrine:** [§213.c — DO-Block Snapshot Isolation](./DOCTRINE_213_MIGRATION_SHAPE_DISCIPLINE.md)
**Method:** AST-lite grep of every migration containing `DO $$`; two-pass classifier:
  1. Coarse pass — DO blocks containing BOTH DDL AND (`information_schema` | `pg_catalog` | `RAISE EXCEPTION`).
  2. Precision pass — DDL statement followed later in same DO block by a query against `pg_constraint | pg_proc | pg_trigger | pg_policies | information_schema.{table_constraints,check_constraints,columns,routines,triggers}`.

## Results

**Total migrations scanned:** 332
**Migrations using `DO $$`:** 50
**Coarse-pass candidates:** 3
**Precision-pass candidates:** 1 file (4 hits)
**True §213.c violations found:** **0**

## Candidate Triage

### 1. `20251015035432_...sql` (business hierarchy fields) — SAFE
Pattern: chain of `IF NOT EXISTS (SELECT information_schema.columns …) THEN ALTER TABLE ADD COLUMN`.
Each guard checks a **different** column than the ALTER that follows it. These are **preflight idempotency guards**, not post-mutation validation of prior DDL. No snapshot dependency on uncommitted state in the same block.

### 2. `20251020021629_...sql` (funding_offers columns) — SAFE
Same preflight-guard pattern as #1. One guarded `ALTER TABLE ADD COLUMN` batch. No cross-column re-checks.

### 3. `20260630042353_...sql` (soft-subagent scanner) — SAFE
DO block only wraps `PERFORM cron.unschedule(...)` with an `EXCEPTION WHEN OTHERS THEN NULL` handler. The `RAISE EXCEPTION` matches live inside a `CREATE FUNCTION` **body definition** (`revoke_platform_access`), which is runtime code, not migration-time DDL validation.

## Precision-Pass False Positives

The 4 hits in `20251015035432_...sql` all match the same `ALTER ADD COLUMN A → IF NOT EXISTS check for COLUMN B` chain. The second `IF NOT EXISTS` queries `information_schema.columns` for a **different** column than the prior ALTER, so snapshot visibility of the just-added column is irrelevant.

## Conclusion

Zero prior migrations violate §213.c. The Ship P.0.1.b Step 3 first-attempt failure that motivated §213.c appears to be the **only** occurrence in project history of a DO block performing post-mutation validation of its own DDL. Doctrine is codified prospectively; no remediation migrations required.

## Residual Risk Notes

The preflight-guard pattern in #1/#2 remains safe **only** while each guard/ALTER pair references a distinct column. Future edits that add a re-check on the same column within the same DO block would trip snapshot isolation. §213.c-aware review should flag any DO block containing two `information_schema.columns` queries against the same `table_name`+`column_name` pair.

## Audit Reproducibility

```bash
# Coarse pass
for f in $(rg -l "^\s*DO \$" supabase/migrations/); do
  awk 'BEGIN{i=0;d=0;v=0;s=0}
    /^\s*DO \$\$/{i=1;d=0;v=0;s=NR;next}
    i && /^\s*\$\$/{if(d&&v)print FILENAME":"s"-"NR;i=0;next}
    i{ if(/EXECUTE|ALTER TABLE|CREATE (OR REPLACE )?(TABLE|FUNCTION|TRIGGER|POLICY|INDEX|CONSTRAINT)|DROP (TABLE|COLUMN|CONSTRAINT|FUNCTION|TRIGGER|POLICY)|ADD (COLUMN|CONSTRAINT)/)d=1
         if(/information_schema|pg_constraint|pg_proc|pg_attribute|pg_class|pg_policies|RAISE EXCEPTION/)v=1 }
  ' "$f"
done
```
